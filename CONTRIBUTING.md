# Contributing to VortX

Thanks for helping build VortX. Contributions are very welcome, and the bar to clear is low: community PRs have been merged within hours of opening.

VortX is one app today (the native Apple client) with more platforms on the way (Android, desktop, and beyond) on a shared engine. This is the general guide for every repository in the organization; each repo's own README has the build instructions and platform specifics.

## How to contribute

1. **Open an issue or discussion first** for anything non-trivial, so we can agree on the approach before you build it. Small, obvious fixes can go straight to a PR.
2. **Fork, branch, and build** following the target repo's README. Keep changes focused: one concern per PR.
3. **Match the existing style** of the code you are touching. Don't reformat or refactor unrelated code in the same PR.
4. **Open the PR** with a clear description of what changed and why, and how you tested it. Screenshots or a short screen recording help for anything visual.

A PR that builds and does one thing well gets reviewed and merged fast.

## What we value

- Clear, focused changes over large sweeping ones.
- Real fixes at the root cause, not patches over symptoms.
- Respect for users: VortX is a client that plays what the user supplies. It hosts no content and bundles none, and contributions keep it that way.

## Conduct

Be decent, and critique code rather than people. See the [Code of Conduct](CODE_OF_CONDUCT.md).

## Security

Found a vulnerability? Please report it privately (see [SECURITY.md](SECURITY.md)), not in a public issue.
